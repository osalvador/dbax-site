Introduction to dbax

dbax (Data Base Application eXpress) is an Open Source Framework for developing Web applications with SQL, PL/SQL and HTML. Using a web browser and SQL tool like SQL Developer or Toad, you can develop and deploy professional Web-based applications for desktops and mobile devices.

dbax is community supported and can be installed in all editions of the Oracle Database. dbax runs within the Oracle Database allowing your applications to scale to meet your largest user communities.

In this technical overview you can see the major components and features of dbax. It describes how easily and quickly you can build beautiful, secure, and scalable Web applications for any devices.

dbax Overview

dbax is an Application Framework

dbax is a toolkit for people who build web applications using Oracle and PL/SQL. Its goal is to enable you to develop projects much faster than you could if you were writing code from scratch, by providing a rich set of libraries or APIs, for commonly needed tasks, as well as a simple interface to access these libraries. dbax lets you focus on your project by minimizing the amount of code needed for a given task.

dbax is Free

dbax is licensed under the LGPL license so you can use it however you please. For more information please read the license agreement.

dabx is Light Weight

Truly light weight. The core system requires only a few very small PL/SQL Packages or libraries. This is in stark contrast to many frameworks that require significantly more resources. Additional libraries are loaded dynamically upon request, based on your needs for a given process, so the base system is very lean and quite fast.

dabx is Fast

dbax has been developed on top of PL/SQL Web Toolkit. Since version 9.2 of the Oracle Database, PL/SQL Web Toolkit is available as a no-cost option and installed by default. PL/SQL Web Toolkit has great performance, so much that Oracle has developed several of its solutions on this technology.

dabx uses M-V-C

dbax uses the Model-View-Controller approach, which allows great separation between logic and presentation. This is particularly good for projects in which designers are working with your template files, as the code these files contain will be minimized.

MVC pattern has become something of a standard in the design of modern web applications. And for good reason. Most web application code falls under one of the following three categories: presentation, business logic, and data access. The MVC pattern models this separation of concerns well. The end result is that your presentation code can be consolidated in one part of your application with your business logic in another and your data access code in yet another. Many developers have found this well-defined separation indispensable for keeping their code organized, especially when more than one developer is working on the same application.


dabx Generates Clean URLs

The URLs generated by dbax are clean and search-engine friendly. Rather than using the standard "query string" approach to URLs that is synonymous with dynamic systems, dbax uses a segment-based approach:

dbax Includes a Template Engine

A template engine is needed to keep interaction with the HTML views, therefore DBAX comes with an easy template engine with a very simple syntax.

For example, this simple code: 

<ul>
 <?dbax
   for c1 in (select   username from all_users)
   loop
      p ('<li>' || c1.username || '</li>');
   end loop;
 ?>
</ul>

Will produce the following result: 

<ul>
 <li>XS$NULL</li><li>HR</li><li>MDSYS</li><li>ANONYMOUS</li><li>XDB</li><li>CTXSYS</li><li>OUTLN</li><li>SYSTEM</li><li>SYS</li><li>DBAX</li>
</ul>

We are working hard to incorporate the new tePLSQL template engine, which is compatible with Oracle PSP. Including tePLSQL in dbax, pl/sql developers will not have to learn a new template language and the applications developed with Oracle PSP will be compatible with dbax

Architecture Overview

dbax utilizes a simple architecture where pages are dynamically generated using a template engine and MVC pattern. Controllers and Models are pl/sql packages that must be compiled in the application schema, not in the dbax schema. Once fully installed and configured, the Uniform Resource Locator (URL) will be defined for access to dbax console. Developers requires a Web browser and SQL client software to build, develop and deploy dbax applications.

All dbax components except Controllers and Models are stored in metadata tables which hold the application definitions. When developers create, modify, or delete any component the changes will be shown immediately. The dbax Console allows the developer to quickly navigate back into the development environment to make additional enhancements as required. This ability to rapidly iterate an application greatly enhances developer productivity.

Single instance. Perfect for VPS cloud infrastructure or development environment.

Multiple instances. Perfect for Enterprise infrastructure with multiple Oracle Databases instances, Oracle RAC or Exadata.

The dbax architecture requires some Web server to proxy requests between a client Web browser and the dbax engine. The Web server options are:

  • Oracle REST Data Services: Oracle REST Data Services (ORDS) is a Java based, free tool that runs in JEE container such as Oracle WebLogic Server, Oracle Glassfish Server, and Apache Tomcat.
  • Embedded PL/SQL Gateway: The Embedded PL/SQL Gateway (EPG) runs in the Oracle XML DB Protocol Server within the Oracle Database and includes the core features of mod_plsql.
  • Oracle HTTP Server: The Oracle HTTP Server (Apache) with mod_plsql plugin can be placed on the same physical machine as the database, or on a separate physical machine. Note: mod_plsql is deprecated as of Oracle HTTP Server 12c (12.1.3). Oracle recommends using Oracle REST Data Services instead.

Web Console

dbax Console - Dashboard

dbax console is an intuitive web application (made also with dbax) that offers a friendly user experience for developers. Is the primary tool for configuration, monitoring and management dbax applications. dbax not contains any configuration files since all this management is done from the web console.

This section describes the main features of the dbax console.

Properties and Settings

dbax Console - Application Settings

dbax Console - Application Properties

Settings

Developers and administrators of dbax applications can change the settings of an application including:

  • enable/disable application. If the application is disabled it will stop responding to requests from users immediately.
  • Change the Access Contror. dbax application can be public or private. When the application is private, users must login with user and password.
  • The user authentication can be performed against a Authentication Scheme (LDAP, Active Directory, local users …).

Properties

Properties are global variables defined in dbax console and are accessible from any application component; views, controllers and models.They are key-value pairs with live changes. By default, dbax adds some properties to new applications:

  • log_level that defines the level of log.
  • resources_url that defines the URL of the resources of this application.
  • base_path the base URL for access to the application.
  • session_cookie_name the name of the cookie session.

URL routing

dbax Console - Routing

dbax has a very practical system by which the URL of the web applications are routed to methods and controllers, or even to views directly. With this system we get search engine friendly URLs, running on pieces of code which also we send all kinds of parameters.

All the power of regular expressions we can apply to dbax routes to create such complex rules as we want:

URL Pattern:

customers/([[:digit:]]+)/orders/([[:digit:]]+)/type/([[:alpha:]]+)

Controller or Method:

pk_c_dbax_crm.orders(\1,\2,'\3')

Test URL:

customers/33245/orders/8769/type/A

Will produce the following procedure call:

pk_c_dbax_crm.orders(33245,8769,'A')

Even you can create case insensitive routes using match_parameter and advanced regex

URL Pattern:

customers/([[:digit:]]+)/orders/([[:digit:]]+)/type/([[:alpha:]]+)@1,1,i

Calling controller using positional parameter association: 

pk_c_dbax_crm.orders(p_customer_id=>\1 , p_order_id=>\2 , p_order_type=>'\3')

Test URL:

CUSTOMERS/33245/Orders/8769/tYPe/A

Will produce the following procedure call:

pk_c_dbax_crm.orders(p_customer_id=>33245 , p_order_id=>8769 , p_order_type=>'A')

The dbax routing uses Oracle regular expressions:

  • Oracle REGEX
  • Oracle REGEXP_INSTR
  • Oracle REGEXP_REPLACE

Web Browser Development

dbax Console - Editing View

dbax view code editor provides HTML validation with inline errors, auto completion, syntax highlighting, search and replace with regex support, complete with undo and redo support.

Logging and tracing

dbax Console - Logging and tracing

In dbax console you can review the logs and traces that have been automatically created in each user request. Use it to identify software problems or live monitor.

In addition through a simple API you can create your custom logs:

dbax_log.debug(‘This is debug message’);
dbax_log.trace(‘This is trace message’);
dbax_log.warn(‘This is warning message’);
dbax_log.error(‘This is error message’);

You can even search within the text log by using Oracle Text CONTAINS Query Operators for advanced filter.

Security

Application developers are responsible for developing secure applications. Defining the appropriate authentication scheme (that is, establishing a user's identity) for an application is critical to ensure only appropriate users can log into the application. As a developer, you must determine which applications do not require authentication (also known as public applications). You should also define authorization schemes to easily define access to the application.


User Authentication

dbax Console - Users Accounts

dbax Console - User Profile

You can define the type of authentication you want to use to validate users:

  • LDAP
  • Active Directory
  • Local Users

From the dbax console you can easily create local users and manage their passwords. And set the authentication scheme for the application.

RBAC Authorization

dbax Console - Permissions assigned to Role

dbax Console - Role Assigned to Users

Role-Based Access Control (RBAC) provides a simple yet powerful centralized access control. The components of RBAC such as role-permissions and user-role relationships make it simple to do user assignments.

User Session State

Session support in dbax consists of a way to preserve certain data across subsequent accesses. User session are stored in the server side. A visitor accessing your web site is assigned a unique id, the so-called session id. The session's data is stored on the database and its session id is stored in a cookie. The session support allows you to store data between requests.

Fast Development

dbax includes numerous features to make your development be as agile and quick as possible. Do not reinvent the wheel, all repetitive tasks that a framework should offer are available in dbax.

Live changes

Any changes you make to any component of dbax, either from the dbax console, as in a controller or model, will be visible at that moment by the users of your application.

You will not have to undeploy your applications or restart any server. This feature accelerates the development of applications because developers do not have to wait for the services restart to review changes, increasing their productivity.

Scaffolding

When you create a new application in dbax you can select an application on which to base as a template. The properties, routes and views that you have created in your template will be copied to your new application.

In addition to this, if the application is private the controllers required for log in users is automatically created.

By default, dbax includes two templates, one based on Bootstrap and another based on Patternfly. Both templates are responsive and focused on creating web applications.

Code Generators (tapiGen2)

With dbax installation, the code generator tapiGen2 is included, with which you can easily create TableAPIs as models for access to their data models.

Again this feature increases the productivity of developers, as well as the maintainability and security of your code.

APIs

dbax includes numerous APIs to facilitate common development tasks. Some of these APIs are:

  • DBAX_COOKIE: for managing user cookies.
  • DBAX_DATATABLE: standardizes the use of Datatable (jQuery Table Plugin) in dbax.
  • DBAX_DOCUMENT: for processing files and documents.
  • DBAX_EXCEPTION: for managing exceptions.
  • DBAX_FILE_PARSER: for parsing files.
  • DBAX_LDAP: for validation and access to LDAP.
  • DBAX_LOG: for recording traces and logs.
  • DBAX_SECURITY: for safety management.
  • DBAX_SSESSION: for managing sessions and user session data.
  • DBAX_UTILS: dbax several utilities.
  • JSON_UTIL: for creating and parsing JSON documents.
  • XLSX_BUILDER: for creating and parsing XLSX document files.
